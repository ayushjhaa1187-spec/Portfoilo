## 2024-05-24 - Timing Attacks in Authentication Middleware
**Vulnerability:** The API lacked proper authentication checks and even when implementing custom authentication using strings, straightforward string comparison (e.g. `===`) introduces a timing attack vulnerability that allows an attacker to guess the API key character-by-character based on server response time.
**Learning:** For custom API key authentication, it is necessary to prevent timing attacks. Node.js `crypto.timingSafeEqual` prevents this, but requires `Buffer` objects. More crucially, passing variable length, user-controlled strings (like headers) into comparison functions or simple hashing methods (like bcrypt, scrypt) could still be vulnerable to length-extension attacks or algorithmic DoS. Using a fast, non-salting hash like SHA-256 to normalize the length and format to a fixed-size `Buffer` *before* calling `crypto.timingSafeEqual` is a highly effective pattern to avoid these pitfalls.
**Prevention:** Always use `crypto.timingSafeEqual` with inputs that have been normalized in length and type using `crypto.createHash('sha256').update(string).digest()` for API key and token comparisons.
