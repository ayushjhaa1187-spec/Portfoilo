## 2024-05-24 - Secure API Key Comparison
**Vulnerability:** Missing authentication on the `POST /api/projects` endpoint allowed unauthorized project creation. Implementing a naive string comparison (`apiKey === process.env.ADMIN_API_KEY`) to fix this would have introduced a timing attack vulnerability, where an attacker could deduce the key length and characters based on the response time.
**Learning:** `crypto.timingSafeEqual` is the correct method for comparing secrets securely, but it throws an error if the buffers are of unequal length. This can cause an application crash (DoS) if an attacker intentionally sends a string of a different length. Pre-hashing the inputs (e.g., using `crypto.createHash('sha256').update(string).digest()`) normalizes the length to a fixed size before comparison, preventing both timing attacks and unequal-length crashes. Using synchronous hashing functions like `crypto.scryptSync` should be avoided for simple comparisons to prevent algorithmic DoS.
**Prevention:** Always use `crypto.timingSafeEqual` with pre-hashed inputs when comparing secrets or API keys in authentication middleware. Never use standard equality operators (`===` or `==`) for sensitive string comparisons.
